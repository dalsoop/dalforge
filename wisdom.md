# Wisdom

5개 레포 운영에서 얻은 교훈. 반복하지 말아야 할 안티패턴과 검증된 패턴을 정리한다.

---

## Anti-Patterns

### 1. 대형 파일 시스템 프롬프트 로드 -- 토큰 폭발

시스템 프롬프트에 파일 전체를 임베드하면 토큰 소비가 기하급수적으로 증가한다.
필요한 부분만 lazy-load하거나, 요약본을 프롬프트에 넣고 원본은 필요 시 참조하는 구조로 전환해야 한다.

> PR #307

### 2. isRetryable 키워드만으로 circuit breaker 판단

에러 메시지의 키워드 매칭만으로 재시도 여부를 결정하면, 예상치 못한 에러에 무한 재시도가 발생한다.
HTTP 상태 코드, 에러 타입 등 구조화된 조건과 max-retry 상한을 조합해야 한다.

> PR #308

### 3. .dal/ 변경만 있을 때 브랜치+push 누적

`.dal/` 디렉토리만 변경된 경우에도 브랜치 생성과 push가 실행되면, 의미 없는 브랜치가 계속 쌓인다.
`.dal/` 전용 변경은 브랜치/커밋/push를 전부 스킵하도록 auto-git-workflow에서 필터링해야 한다.

> PR #356

### 4. api/message 봇 토큰으로 발송하면 자기 메시지 스킵

봇이 보낸 메시지를 봇 자신이 다시 수신하면 무한 루프에 빠지거나 메시지를 무시하게 된다.
발신자 ID를 체크하여 자기 메시지를 명시적으로 필터링하는 로직이 필요하다.

> PR #329

### 5. leader가 직접 코드 수정+커밋

leader는 라우터/판단자 역할이다. 직접 코드를 수정하고 커밋하면 리뷰 체계가 무너진다.
코드 변경은 반드시 dev에게 assign하고, leader는 PR 생성/머지만 담당해야 한다.

> PR #334, #335

### 6. localdalRoot를 cwd 기반으로 결정 -- 잘못된 mount

현재 작업 디렉토리(cwd)를 기준으로 `.dal/` 경로를 결정하면, 실행 위치에 따라 잘못된 디렉토리를 마운트한다.
git root 또는 명시적 환경변수를 기준으로 절대 경로를 결정해야 한다.

> PR #330

### 7. nohup 데몬

`nohup`으로 데몬을 띄우면 프로세스 관리(재시작, 로그, 상태 확인)가 불투명해진다.
컨테이너 기반 실행 또는 systemd 등 프로세스 매니저를 사용하여 라이프사이클을 명확히 관리해야 한다.

### 8. wake마다 새 봇 생성

멤버를 wake할 때마다 새로운 봇 인스턴스를 생성하면 리소스가 낭비되고, 이전 세션의 컨텍스트가 유실된다.
봇 인스턴스는 재사용하고, 세션 상태는 외부 저장소에서 관리해야 한다.

## Patterns

### 1. .dal/ read-only overlay mount

`.dal/` 디렉토리를 read-only overlay로 마운트하면, 멤버가 설정 파일을 실수로 수정하는 것을 방지하면서도 최신 설정을 참조할 수 있다.
변경은 호스트 측에서만 이루어지고, 컨테이너 내부에서는 읽기 전용으로 접근한다.

### 2. state 디렉토리를 git 밖에 배치

런타임 상태 파일(PID, 소켓, 임시 데이터)을 git 저장소 밖에 두면, 불필요한 diff와 커밋 충돌을 방지할 수 있다.
`.dal/data/` 등 상태 디렉토리는 `.gitignore`에 등록하거나 아예 저장소 외부 경로를 사용한다.

### 3. scribe dal로 문서 전담

문서 작성/갱신을 전담하는 scribe dal을 두면, 다른 멤버가 문서 작업에 시간을 빼앗기지 않고 본업에 집중할 수 있다.
CHANGELOG, README, 릴리스 노트 등 반복적 문서 작업은 scribe에게 assign한다.
